GUÍA TÉCNICA PARA ESTUDIANTES

ai-service

Este documento explica cómo funciona el microservicio de inteligencia artificial de SlideHub: qué hace, cómo fluye la información entre los modelos, y cómo se almacena el resultado. Está escrito para alguien que ya sabe programar pero que ve este código por primera vez.

¿QUÉ ES EL AI-SERVICE?

ai-service es uno de los cuatro microservicios de SlideHub. Su única responsabilidad es coordinar llamadas a modelos de IA externos y persistir los resultados en MongoDB para no tener que regenerarlos cada vez.

Recibe
  • URLs de imágenes de slides (alojadas en S3)
  • URLs de repositorios GitHub
  • Archivos de audio de participantes
  • Plataforma de despliegue objetivo
Produce
  • Notas estructuradas por slide
  • Análisis técnico del repositorio
  • Dockerfiles y guías de despliegue
  • Respuestas en audio a preguntas del público
¿Por qué separarlo en su propio servicio?
Las llamadas a Gemini y Groq tardan entre 2 y 15 segundos y tienen costo por uso. Al aislarlas en un servicio propio: el resto de la app no se bloquea esperando, se puede escalar independientemente, y los resultados se cachean en MongoDB para evitar pagar dos veces por la misma generación.
ARQUITECTURA DEL SISTEMA

SlideHub tiene cuatro microservicios. Todos los pedidos externos llegan primero al gateway-service, que decide a qué servicio enrutar. El ai-service nunca recibe tráfico directo del navegador.

LOS MODELOS DE IA

El servicio usa dos proveedores distintos, cada uno con sus fortalezas. Ambos se consumen via HTTP con WebClient — sin SDKs oficiales.

Google Gemini gemini-1.5-flash

Modelo multimodal: puede recibir texto e imágenes en la misma solicitud. Se usa cuando hay que "ver" el contenido visual de un slide.

Úsado para:
  • Analizar imagen de un slide (Vision)
  • Extraer contexto técnico de un repositorio
  • Análisis profundo del stack tecnológico
Groq llama3-8b + whisper

Inferencia ultrarrápida (tokens/seg muy altos) para generación de texto estructurado y transcripción de audio con Whisper.

Úsado para:
  • Generar notas del presentador (JSON)
  • Transcribir audio (Whisper large-v3)
  • Generar Dockerfiles y guías de deploy
  • Responder preguntas del público
Importante para la comprensión del código: Groq expone una API compatible con OpenAI (/openai/v1/chat/completions). Esto significa que aunque usamos Groq, el formato de request y response es idéntico al de OpenAI. Si alguna vez migramos a OpenAI, solo cambia la URL y el API key.

Paso 1 resolveSlideDescription()

Si el request incluye una imagen (Base64 o URL de S3), se descarga y se envía a Gemini Vision para que describa qué ve: título, conceptos técnicos visibles, tipo de diagrama, etc. Si no hay imagen, se usa el texto descriptivo enviado en el campo slideContext como fallback.

Paso 2 extractRepoContext()

Con la descripción del slide del paso 1, Gemini visita el repositorio GitHub indicado y extrae el contenido más relevante para ese slide concreto: archivos relacionados, patrones de código, configuraciones, etc. Si no hay repositorio, este paso se omite.

Paso 3 generateNote()

Groq recibe el contexto de los dos pasos anteriores y genera una nota estructurada en formato JSON con: título, puntos clave, tiempo sugerido, frases clave y tags de demo. Groq tiene restricción explícita de responder solo JSON válido.

La nota resultante tiene esta forma en MongoDB:

{ "presentationId": "abc-123", "slideNumber": 3, "title": "Arquitectura de microservicios", "points": [ "Cada servicio tiene una responsabilidad única", "Comunicación via HTTP REST con WebClient", "El gateway centraliza el enrutamiento" ], "suggestedTime": "~3 min", "keyPhrases": ["principio de responsabilidad única", "gateway pattern"], "demoTags": ["demo-gateway-routing"] }
¿Qué pasa si ya existe una nota para ese slide?
El servicio hace un upsert: busca si ya existe una nota con el mismo presentationId + slideNumber. Si existe, la reemplaza preservando el mismo _id de MongoDB. Si no existe, crea una nueva.

Gemini analiza el repositorio y devuelve un JSON con toda la información técnica relevante:

{ "language": "Java", "framework": "Spring Boot", "buildSystem": "Maven", "technologies": ["Redis", "PostgreSQL", "Thymeleaf", "Docker"], "databases": ["PostgreSQL", "Redis"], "ports": [8080, 8081, 8082, 8083], "environment": ["DATABASE_URL", "REDIS_HOST", "GEMINI_API_KEY"], "summary": "Sistema multi-pantalla de presentaciones con microservicios", "structure": "4 microservicios: gateway, state, ui, ai", "deploymentHints": "Usar Render con Web Services separados, Redis addon" }
Caché inteligente: Si el mismo repositorio se analiza varias veces (distintos usuarios, distintas presentaciones), Gemini solo se llama una vez. Después, todos leen de MongoDB. Para forzar un nuevo análisis existe el endpoint /api/ai/analyze-repo/refresh que borra el caché y recalcula.

La respuesta incluye:

{ "success": true, "transcription": "¿Cómo funciona el gateway?", "answer": "El gateway centraliza el enrutamiento..." }
El contexto hace la diferencia: Groq no solo responde la pregunta transcrita, sino que tiene en cuenta qué slide se está mostrando y el contenido técnico del repositorio. Esto permite respuestas mucho más precisas que un chatbot genérico.
BASE DE DATOS — MONGODB

El ai-service usa MongoDB exclusivamente. Se eligió MongoDB (en lugar de PostgreSQL) porque los datos generados por IA son documentos semi-estructurados: listas de longitud variable, campos opcionales, texto largo. Encajan mejor en un modelo de documentos que en tablas relacionales.

Colección presenter_notes

Índice único compuesto en (presentationId, slideNumber)

suggestedTimeStringTiempo estimado, ej: "~2 min"
keyPhrasesList<String>Frases técnicas clave del slide
demoTagsList<String>Tags de demos a mostrar en este slide

REFERENCIA DE ENDPOINTS

Todos los endpoints son accesibles a través del gateway-service en el puerto 8080. Los que generan IA tienen rate limiting activo.

Notas del presentador
POST /api/ai/notes/generate Genera nota para un slide individual (Gemini Vision + Groq)
POST /api/ai/notes/generate-all Genera notas para todos los slides de una presentación
GET /api/ai/notes/{presentationId} Lista todas las notas de una presentación
GET /api/ai/notes/{presentationId}/{slideNumber} Nota de un slide concreto (204 si no existe)
DELETE /api/ai/notes/{presentationId} Elimina todas las notas de una presentación
Análisis de repositorio
POST /api/ai/analyze-repo Analiza el repo (usa caché MongoDB si ya existe)
POST /api/ai/analyze-repo/refresh Fuerza nuevo análisis ignorando el caché
Deploy Tutor
POST /api/ai/deploy/analyze Analiza repo para despliegue (misma lógica que analyze-repo)
POST /api/ai/deploy/dockerfile Genera Dockerfile optimizado con Groq
POST /api/ai/deploy/guide Genera guía de despliegue para Render/Vercel/Netlify
POST /api/ai/deploy/guide/refresh Regenera la guía borrando el caché
Asistente de audio
POST /api/ai/assist/audio Transcribe audio y genera respuesta contextual (multipart)
Rate limiting activo en el gateway: Los endpoints de IA tienen límites de 5–6 solicitudes por minuto por IP para controlar el costo de las APIs. Si una solicitud supera el límite, el gateway devuelve 429 Too Many Requests con el header Retry-After indicando cuántos segundos esperar.
